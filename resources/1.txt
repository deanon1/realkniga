Диаграмма вариантов использования является одной из ключевых моделей в языке моделирования UML[1]. Она предназначена для высокоуровневого описания функциональности системы через взаимодействие внешних субъектов (актеров) с программным продуктом. Данная модель позволяет четко определить границы системы, её возможности и основные сценарии поведения без детализации внутренней реализации.
Актеры системы:
	пользователь – основной субъект, который взаимодействует с графическим интерфейсом приложения для ведения учета финансов, анализа трат и управления дисконтными картами;
	API НБРБ – внешняя информационная система, поставляющая актуальные курсы валют по запросу приложения для корректного отображения баланса в различных эквивалентах.
Основные варианты использования, представленные на диаграмме, сгруппированы по функциональным блокам:
	аутентификация: обеспечение безопасности данных через вход по ПИН-коду или с использованием биометрических данных;
	главный экран: оперативный просмотр последних транзакций, текущего баланса и актуальных курсов валют;
	управление транзакциями: добавление доходов и расходов вручную, редактирование или удаление записей. Использование функции сканирования QR-кодов чеков является расширением сценария добавления операции;
	аналитика расходов: изучение графиков по категориям, просмотр детальных курсов валют и экспорт ежемесячной финансовой отчетности в формат PDF;
	кошелек скидочных карт: просмотр деталей и управление базой карт. Сценарий добавления карты дополняется возможностью сканирования штрих-кода с физического носителя;
	управление категориями: создание пользовательских категорий расходов и установка индивидуальных лимитов бюджета для контроля перерасхода средств;
	настройки приложения: персонализация интерфейса, выбор основной валюты, настройка ежедневных напоминаний и управление параметрами безопасности.
Взаимодействие между актерами и системой строится на принципе событийности: действия пользователя инициируют запросы к локальной базе данных SQLite или внешнему API. Результаты обработки данных мгновенно визуализируются в интерфейсе в виде обновленных графиков, списков транзакций или активных дисконтных кодов.


Логическая структура базы данных, представленная на рисунке 1, включает три основные сущности: transactions (транзакции), categories (категории) и discount_cards (скидочные карты).
Сущность «transactions» предназначена для регистрации всех финансовых операций. Её структура включает следующие атрибуты:
	id – уникальный идентификатор записи (INTEGER PRIMARY KEY);
	amount – сумма операции (REAL);
	category – текстовое поле, связывающее транзакцию с конкретной категорией;
	date – дата и время совершения операции в строковом формате (ISO 8601);
	isIncome – целочисленный флаг (0 или 1), определяющий тип операции (доход или расход);
	comment – произвольное текстовое описание транзакции.
Сущность «categories» служит для систематизации трат и управления бюджетом. Она имеет связь с таблицей транзакций по полю name. Атрибуты таблицы:
	id – первичный ключ записи;
	name – уникальное наименование категории (например, «Еда», «Транспорт»);
	iconCode – целочисленный код иконки из набора Material Design;
	isDefault – флаг, отделяющий системные категории от созданных пользователем;
	budgetLimit – числовой показатель ежемесячного лимита для данной категории.
Сущность «discount_cards» инкапсулирует информацию о цифровых копиях скидочных карт пользователя. В её состав входят:
	id – уникальный идентификатор карты;
	storeName – текстовое название торговой сети или магазина;
	code – оцифрованные данные штрих-кода;
	format – стандарт кодирования;
	color – целочисленное значение цвета для визуализации карты в интерфейсе.
Взаимодействие с базой данных реализовано через класс DatabaseService. Он инкапсулирует в себе методы для выполнения всех базовых CRUD-операций: создание новых записей, чтение списков с сортировкой, обновление параметров и удаление устаревших данных.
Обновление пользовательского интерфейса при изменении данных в БД реализовано на основе паттерна Provider и механизма ChangeNotifier. При любом изменении в локальном хранилище (например, добавлении новой покупки через QR-сканер) срабатывает метод notifyListeners(), который инициирует автоматическую перерисовку соответствующих экранов и графиков, обеспечивая пользователя актуальной информацией о состоянии бюджета в режиме реального времени. Диаграмма классов является фундаментальным инструментом проектирования, который визуализирует статическую архитектуру программного комплекса «MyFinance». Она позволяет детально разобрать внутреннее устройство приложения, определяя состав основных программных модулей, их внутренние атрибуты, методы и логические связи. Архитектура системы построена на принципах разделения ответственности, где выделены слои моделей данных, управления состоянием и вспомогательных сервисов, что гарантирует масштабируемость и стабильность работы продукта[1].
Основные компоненты системы, отраженные на диаграмме:
	блок моделей (Models): классы TransactionModel, CategoryModel и DiscountCardModel выступают в роли контейнеров для хранения структурированной информации. Они инкапсулируют в себе параметры финансовых операций (суммы, даты, категории), настройки лимитов бюджета и данные о скидочных картах пользователя. Класс CurrencyRate используется для временного хранения актуальных котировок, полученных извне;
	провайдеры управления состоянием (Providers): классы TransactionProvider и DiscountCardProvider являются «мозгом» приложения. Они реализуют паттерн ChangeNotifier, обеспечивая реактивную связь между базой данных и интерфейсом. Эти классы отвечают за логику загрузки данных, фильтрацию транзакций, расчет текущего баланса и инициируют обновление экранов при любых изменениях;
	DatabaseService: центральный сервисный класс, реализованный в виде одиночки (Singleton). Он обеспечивает прямой доступ к локальному хранилищу SQLite, выполняя низкоуровневые SQL-запросы для сохранения и извлечения информации.
Блок сервисов и утилит:
	CurrencyService: отвечает за сетевое взаимодействие с API НБРБ, выполняя парсинг данных для актуализации валютных курсов в приложении;
	PdfService: реализует функционал формирования финансовых отчетов, преобразуя списки транзакций в структурированный формат PDF для экспорта;
	NotificationService: управляет системой локальных уведомлений, планируя ежедневные напоминания о необходимости внесения записей.
Логические взаимосвязи, представленные на диаграмме, демонстрируют архитектурную целостность приложения: от фиксации внешнего события до обработки данных сервисами и их итогового закрепления в локальной базе данных


Диаграмма последовательности детально отображает динамику взаимодействия компонентов приложения «MyFinance» во времени: обмен сообщениями между пользователем, графическим интерфейсом и сервисным слоем. Рассматриваются основные сценарии – от запуска до автоматического обновления экрана после операции.
При старте загружаются пользовательские настройки и актуализируются финансовые данные. CurrencyService через API Нацбанка РБ получает свежие котировки валют. Обработанные значения передаются в интерфейс: пользователь видит обновленный баланс и курсы в виджете главного экрана.
Регистрация операций возможна двумя способами. При ручном вводе пользователь заполняет форму AddTransactionSheet, данные передаются в службу управления БД. При QR-сканировании камера захватывает код чека, система делает HTTP-запрос к серверу СККО для извлечения деталей покупки. Алгоритмы парсят HTML, автоматически подставляя сумму и дату, ускоряя учет и снижая ошибки.
После подтверждения операции бизнес-логика сохраняет данные в SQLite. TransactionProvider через DatabaseService фиксирует запись и запускает пересчет баланса (разница доходов и расходов). Если в настройках включены уведомления, NotificationService планирует ежедневные напоминания.
На финальном этапе после сохранения транзакции срабатывает механизм notifyListeners(), перерисовывая виджеты. Пользователь мгновенно видит новую операцию в списке и изменения на графиках, а подтверждение сопровождается анимацией конфетти.


Диаграмма компонентов предназначена для визуализации физической структуры системы, определения программных модулей и существующих между ними зависимостей. Основной целью проектирования данной структуры является обеспечение модульности и независимости слоев бизнес-логики от пользовательского интерфейса, что гарантирует масштабируемость проекта. Архитектура приложения «MyFinance» разделена на логические пакеты, каждый из которых инкапсулирует строго определенную функциональность.
Описание ключевых структурных элементов диаграммы:
	узел исполнения: результирующий файл сборки, включающий в себя скомпилированный код и ресурсы приложения. Точкой входа в логику программы является компонент main.dart, который отвечает за первичную инициализацию базы данных, системных настроек и запуск интерфейса[7,8];
	пакет «core»: содержит базовую инфраструктуру приложения. В данный модуль включены механизмы управления состоянием, сервисы для работы с валютами, PDF-отчетами и уведомлениями, а также вспомогательные утилиты и настройки визуальной темы;
	пакет «data»: отвечает за хранение и извлечение информации. Пакет объединяет в себе модели данных и центральный сервис database_service.dart, обеспечивающий работу с локальной реляционной базой данных SQLite;
	пакет «presentation»: слой пользовательского интерфейса, организованный по принципу разделения на экраны и переиспользуемые виджеты. Он включает модули для мониторинга баланса, управления кошельком дисконтных карт, просмотра финансовой статистики и настройки безопасности. Роль центрального элемента навигации выполняет main_wrapper.dart, координирующий переходы между основными разделами приложения.
Зависимости, указанные на диаграмме, демонстрируют иерархическую структуру: высокоуровневые компоненты интерфейса зависят от логики провайдеров и сервисов, которые, в свою очередь, взаимодействуют с уровнем данных. Использование такой архитектуры позволяет минимизировать связность компонентов, что значительно упрощает тестирование и дальнейшее сопровождение программного продукта «MyFinance».
